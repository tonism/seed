# X.509 leaf parse + verify-vs-WR1 — the contract the 286 port implements

This is the build-against spec for the silent-auto-recertify trust core. The offline oracle
(`x509_verify.py`) is the **authoritative reference**; the 286 module must mirror it byte-for-byte
in *what it accepts and rejects*. A wrong parse accepts a forged cert = false security (the worst
failure), so the rule is **strict DER, positional walk, fail closed**.

Proven offline (run both before trusting any port):

```
python3 tools/x509/x509_verify.py --leaf tools/x509/certs/leaf.der --anchor tools/x509/certs/wr1.der --now 2026-07-21T12:00:00
python3 tools/x509/tamper_matrix.py     # 27 cert vectors + TLV strict-DER unit tests, all green
```

## Trust model (confirmed against the live endpoint, 2026-07-21)

`api.openai.com` is **dual-cert load-balanced**. Forcing the 286's exact profile
(`ECDHE-RSA` + `RSA+SHA256`) returns an **RSA-2048 leaf issued by GTS WR1**; default ECDSA-capable
clients get a P-256 leaf via WE1. The 286 only ever offers ECDHE_RSA, so it always lands in the WR1
RSA family. The current checked leaf is valid from **2026-07-08** to **2026-10-06**; WR1 is
**RSA-2048, valid until 2029-02-20** — the durable anchor. (Risk to flag, not a blocker:
OpenAI now *prefers* ECDSA in general; the RSA leaf survives only because the 286 forces it. If
GTS ever drops the RSA leaf, recertify fails closed and the user re-pins — safe.)

- **Anchor (baked, durable):** WR1's RSA-2048 key → `core/rsa_anchor_wr1.inc` (`wr1_n`),
  generated by `gen-rsa-pinned-key.py --mode anchor`. Re-pin only on WR1 rotation (rare, human).
- **External leaf pin (the fast path):** standard images ship the current leaf as `SEED/LEAF.DER`.
  Later 286+ connects verify that DER against WR1/SAN/date before adopting it into the
  runtime-only `rsa_workspace.inc`.
- **Runtime refresh:** after a verified rotation, Seed may write the captured DER back to
  `SEED/LEAF.DER`. The file is never trusted just because it is on disk.

## The flow (off-race architecture — the @6 unlock, unchanged from the plan)

1. **Pre-socket:** read `SEED/LEAF.DER`, verify it through the same parser/WR1/date path, then adopt
   it into `rsa_n`. Missing or invalid files leave no ready key and force recertification.
2. **In-race (every handshake):** verify the SKE sig against the adopted leaf `rsa_n` — 1 RSA verify,
   fits 6 MHz.
3. **Detect stale:** that verify fails → the leaf rotated (or an attack). Do NOT chain-verify in race.
4. **Off-race recertify** (shown as a dim `> recertify`): take the leaf from the Certificate message,
   run the parse+verify below against `wr1_*`, and on success **adopt** the new leaf (below). Retry.
5. **Refresh:** before the retry opens a new socket, try to write the verified capture as
   `SEED/LEAF.DER`. Write failure is silent; read-only media just recertifies again after rotation.
6. Fail closed: an un-verifiable leaf refuses (`> recertify failed` → DPI), history stays in RAM.

## DER structure (real leaf — offsets are illustrative; the port walks POSITIONALLY, not by offset)

```
Certificate ::= SEQUENCE {                          @0   (0x30 0x82 len)
  tbsCertificate    SEQUENCE { ... },               @4   <-- the SIGNED span: der[tbs_tag .. tbs_end]
  signatureAlgorithm SEQUENCE { OID, NULL },        @1066  must == sha256WithRSAEncryption
  signatureValue    BIT STRING (00 || 256-byte sig) @1081  the RSA signature over SHA-256(tbs_span)
}
TBSCertificate ::= SEQUENCE {
  [0] EXPLICIT version INTEGER (==2, i.e. v3),       required
  serialNumber  INTEGER,
  signature     SEQUENCE { OID, NULL },              must EQUAL the outer signatureAlgorithm
  issuer        Name (SEQUENCE),                     not inspected (the sig-vs-WR1 IS the issuer check)
  validity      SEQUENCE { notBefore, notAfter },    UTCTime "YYMMDDHHMMSSZ" / GeneralizedTime
  subject       Name (SEQUENCE),                     not inspected
  subjectPublicKeyInfo SEQUENCE {                    the leaf key to ADOPT
      SEQUENCE { rsaEncryption OID, NULL },
      BIT STRING (00 || RSAPublicKey SEQ{ INTEGER n, INTEGER e }) },   require RSA-2048, e==65537
  [3] EXPLICIT extensions SEQUENCE OF Extension {
      Extension ::= SEQUENCE { extnID OID, critical BOOLEAN DEFAULT FALSE, extnValue OCTET STRING }
      subjectAltName (OID 2.5.29.17): OCTET STRING wraps SEQUENCE OF GeneralName;
          dNSName == context tag [2] (0x82) primitive IA5String }
}
```

OID byte sequences the port byte-compares (full TLVs):
- sha256WithRSAEncryption `06 09 2A 86 48 86 F7 0D 01 01 0B`
- rsaEncryption           `06 09 2A 86 48 86 F7 0D 01 01 01`
- subjectAltName          `06 03 55 1D 11`
- SHA-256 DigestInfo prefix (19 bytes, already in `p256_module.asm` as `rsa_sha256_digestinfo`):
  `30 31 30 0D 06 09 60 86 48 01 65 03 04 02 01 05 00 04 20`

## Strict-DER reader rules (the false-security crux — every rule has a tamper vector)

- **Length:** short form (<0x80) or long form. Reject indefinite (`0x80`) and reserved (`0xFF`).
  Long form must be **minimal**: value ≥ 0x80 and no leading-zero length octet. Device needs ≤2
  length octets (certs are <64 KB); reject longer.
- **Bounds:** every TLV's content must fit inside its parent. Reject any overrun.
- **No trailing bytes:** the buffer must be exactly consumed at the top level and at each container
  whose contents are fixed (validity = exactly 2 times; RSAPublicKey = exactly n,e; etc).
- **Tag must match** the expected tag at each positional step.
- **INTEGER:** non-negative, minimally encoded (no illegal leading 0x00).
- **BIT STRING** (SPKI key, signatureValue): first content byte must be `0x00` (zero unused bits).

## Verification gates (ALL must pass; order as in `verify_leaf`)

1. `inner signatureAlgorithm == outer signatureAlgorithm == sha256WithRSAEncryption`.
2. **Signature vs the pinned WR1 key:** `m = sig^65537 mod wr1_n` (reuse `rsa_verify`, with WR1's
   constants loaded), then **reconstruct** the full EMSA-PKCS1-v1.5 block `00 01 FF×202 00
   DigestInfo SHA256(tbs_span)` and compare all **256** bytes (the existing `rsa_build_expected_block`
   + memcmp path — no lax DigestInfo re-parse; blocks padding forgeries). The SHA-256 is over the
   exact captured `tbs_span`. The resident K-window does the SHA (the module has none) and hands the
   32-byte hash in, exactly like the SKE path.
3. **Adopt-key sanity:** the leaf SPKI must be RSA-2048 with e==65537 (the device modexp is fixed).
4. **Exactly one** SubjectAltName extension, carrying the **exact** dNSName `api.openai.com`
   (case-insensitive byte compare; wildcards deliberately NOT honored — the pinned host is fixed and
   the real leaf carries the exact name, so wildcard logic is pure attack surface).
5. **Validity:** notBefore ≤ now ≤ notAfter. (Device clock source: the handshake already mixes a
   PIT/tick; a coarse wall-clock is enough for a 90-day window. TBD in the port — see open items.)

Deliberately **out of scope** (documented, non-load-bearing under WR1-pin + exact-host): rejecting
unrecognized *critical* extensions (an attacker with a WR1-issued api.openai.com cert already owns
the host); general duplicate-OID rejection beyond the single-SAN gate.

## Adopt (on a verified rotation) — feasible on the 286, cheap

Install the new leaf as the fast-path pin in RAM (the read-only floppy can't persist it; re-derived
from WR1 each boot after a rotation):
- `rsa_n ← parsed leaf SPKI modulus` (256 bytes, copy).
- `rsa_n0inv ← -n^-1 mod 2^16` (16-bit; ~5-iteration Newton/extended-Euclid — trivial).
- `rsa_r2 ← 2^4096 mod n` (start x = 2^2048 mod n = 2^2048 − n since n's top bit is set; then 2048
  modular doublings `x = 2x mod n`, each a shift + conditional subtract of n; ~well under 1 s @6).
Then retry the handshake → back to the 1-verify fast path against the new `rsa_n`.
